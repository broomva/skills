SECTION 01

Thesis — a thin vertical, not a new orchestrator

kleos-engine is a social content plant bolted onto control loops Broomva already runs in production. The Kleos flywheel (Brand Lock → Show → Distribute → Hook → Sell → Moat) is an observe → decide → act → judge control loop. Symphony already runs exactly that shape for code tickets; Kleos swaps the plant from "a git repo + CI" to "a social graph + retention metrics." Everything else — the scheduler, the eval ledger, the control gate — is reuse.

Concretely: symphony-orchestrator's scheduler.rs already owns a poll tick that "coordinates dispatch, reconciliation, and retries" on an interval. That is the observe→decide→act spine. The only thing it doesn't have is a plant whose state is "what did the audience do to yesterday's posts" and an actuator that talks to platform APIs. Those are the deltas Kleos adds — bounded, named, and budgeted below.

Why this is honest reuse

Swapping the plant is a legitimate vertical: the directive/trace/evaluator interfaces from agentic-control-kernel are plant-agnostic by design. What is not honest is claiming the judge already does variant selection. It does not — see §2.

SECTION 02

The honest self-improvement scoping

This is the review's #1 fix and the spec's most important correction. The draft claimed Kleos's self-improvement was a fork of symphony-egri/batch.rs. The reviewer read the file. It is not a variant selector. It is a pass/fail tally.

What batch.rs actually does (verified, line-by-line)

symphony-egri/src/batch.rs computes one scalar: score = completed / (completed + retrying). When that scalar clears a threshold it increments promoted_count; otherwise discarded_count. That is the entire mechanism — a batch-level success rate with a counter. It has no notion of a variant, no per-arm reward, no attribution of which creative choice drove retention, and no parameter that gets written back into tomorrow's generation.

True self-improvement is unbuilt NEW code, comprising three pieces none of which exist today:

  • A multi-armed bandit — pull arms = creative variants (hook style, lighting signature, caption pattern, posting hour); allocate the next batch's budget toward arms with higher posterior reward.
  • Per-variant retention attribution — join each published asset's variant fingerprint to its measured retention/conversion, so the reward signal is causal, not a batch average.
  • Brand-DNA parameter fold-back — write the winning parameters back into the content-engine-dna prompt skeleton so the next Brand-Lock starts from a measured optimum.

Per CLAUDE.md §Ritual vs Substance: "a phrase that recurs as a discipline must map to a concrete machine-checkable behavior, or it does not count as discipline." Claiming "self-improving" while pointing at a counter is the canonical incantation-vs-control failure — a verbal claim with no causally-independent mechanism (h ⟂ U) behind it. The canon's kleos_check.py hard-codes the gate:

The binding gate

self_improvement.claimed == true ⟹ mechanism_built == true && measured == true, else FAIL. No Kleos surface — buyer-facing or internal — may say "self-improving" or "the loop rewrites tomorrow's content" until the bandit + attribution + fold-back are built AND measured.

Scoping decision

The self-improvement loop is therefore scoped to M2, with its own risk budget — not claimed as M1 reuse. M1 ships the tally (the existing eval ledger, honestly labeled as a batch success-rate gate, which is all it is). The bandit is a research-grade subsystem and gets the elapsed-time and review surface its novelty warrants, not a one-line "fork batch.rs."

The M1 store decision

Because Lago/Anima/Haima are in-flight, kleos-store M1 uses a local embedded store (sled or sqlite) for event-sourced run state. Lago wiring is M2, after the embedded store has proven the event schema. This removes a multi-week cross-subsystem dependency from the critical path of the first shippable runtime.

SECTION 05

The publisher transport reality

This table is the load-bearing engineering constraint of the whole runtime. kleos-publisher is not "post to social" — it is four different transports with four different hard limits, and those limits dictate the architecture (per-platform ceilings in the shield, the X-never-leads-with-a-link rule, TikTok-private-until-audit). Get this wrong and the daemon gets rate-limited, throttled, or banned on Day 1.

RailAPIHard limit (the constraint)Architectural consequence
YouTube Data API v3 ~100 quota units / call; default free quota 10,000/day → ~100 uploads/day ceiling The M1 first rail — generous enough quota that a single rail proves the loop without quota engineering. Account each call against the daily budget in kleos-store.
Instagram Graph API 3-step create container → poll status → publish; hard 25 published / 24h; requires App Review M1-second. The poll-status step makes publish asynchronous — the loop must persist a pending-container state and re-enter, not block. 25/24h is a non-negotiable S-K2 ceiling.
TikTok Content Posting API 6 req / min; SELF_ONLY until the content-posting audit clears Private-until-audit. M1 publishes to the owner's private drafts only; public posting is gated on a manual audit milestone, not a code milestone. Do not block M1 on it.
X v2 pay-per-use $0.015 text-write; $0.20 URL-write penalty The penalty ENFORCES the strategy. A 13× cost for posting a link makes "X = authority layer, never lead with a link" a budget constraint, not just a guideline. M1 = text/threads only.
Why the X penalty is a feature

The canon's "X is the authority layer, not a video dump" rule is usually a discipline an operator has to remember. On X v2 it is self-enforcing: the $0.20 URL-write penalty (13× the text-write cost) means a budget-capped agent structurally cannot spam links. The shield's S-K1 spend cap turns the strategy into arithmetic.

SECTION 06

The governance shield

Every generate/publish action passes a pre-action gate before it touches an API — the same PreToolUse shield pattern that .control/policy.yaml already runs for the workspace, extended with five Kleos shields. The gate verdict is Pass / Modify / Block, evaluated before the action, never after.

S-K1 · spend-cap/day

Total API + model spend bounded per 24h. The X URL-write penalty and Veo/Kling per-second costs are accounted here.

block if Σspend > cap

S-K2 · per-platform ceilings

Each rail's published-count ≤ its API hard limit (IG ≤ 25/24h, TikTok ≤ 6/min, YT ≤ quota).

≤ API hard limit, always

S-K3 · disclosure = 100%

Every asset carries C2PA Content Credentials + platform AI-label as a build step. EU AI Act Art. 50 (live Aug 2 2026).

block if any asset undisclosed

S-K4 · likeness-firewall = 0

Zero real identifiable persons without written consent + license. Synthetic personas / cleared talent only.

block on real-person likeness

S-K5 · brand-safety

Content classifier gate against banned categories before publish; keeps the rail off the YouTube inauthentic-content sweep.

block on unsafe class
Tie-in to policy.yaml

S-K1…S-K5 are added as rows to .control/policy.yaml alongside the workspace's existing S1…S15 shields and G1…G5 gates — same schema, same enforcement surface. Kleos does not invent a governance mechanism; it adds setpoints to the one that already runs. S-K3 and S-K4 are the buyer-facing moat (the "won't get you banned or sued" compliance pillar) and the runtime's own safety rail — the seller (Kleos) is the FTC-scope entity, so the shield hardens Kleos first.

SECTION 07

M1 build plan (gated)

A realistic week-by-week that ships one rail first, defers the in-flight dependencies, and explicitly marks the bandit as M2. This plan does not start until the §00 gate clears — it is the shape of M1, not a schedule to run today.

WEEK 1
Spine + store. Stand up kleos-core against the existing scheduler.rs poll-tick with a content-plan input. Embedded kleos-store (sled/sqlite) with the event schema. No publishing yet — drive kleos-generate (content-engine adapter) to render .mp4 to disk. no Lago — embedded only
WEEK 2
Shield + disclosure. Wire kleos-shield with S-K1…S-K5 into .control/policy.yaml. Build the C2PA + AI-label disclosure step (S-K3) as part of the render pipeline — disclosure is a build step, not an afterthought. Gate every generate behind Pass/Modify/Block. P2 · Gate
WEEK 3
One rail: YouTube. Build kleos-publisher's YouTube Data API v3 transport with hard quota accounting against kleos-store. Actually publish a real asset and screenshot the live upload — reasoning is not validation. P11 · Empirical
WEEK 4
Close the loop (as a tally). kleos-feedback polls YouTube analytics, writes retention into kleos-store, and runs the existing eval ledger tally (honestly labeled — NOT a bandit). The loop is now closed end-to-end on one rail. tally, not variant selection
WEEK 5
IG second, TikTok private, X text-only. IG Graph 3-step transport (async container → poll → publish; S-K2 ≤ 25/24h). TikTok transport publishes SELF_ONLY drafts only (public gated on audit). X v2 transport text/threads only (no URL writes — the $0.20 penalty enforces it). multi-rail
SHIP
Deploy to Railway. P20 cross-model adversarial review before merge (the writer model cannot be the final judge). Deploy the daemon to Railway. The runtime is real iff kleos_check.py passes on its manifest. P20 before merge · P4 pipeline
Explicitly NOT in M1

The self-improvement loop (bandit + per-variant attribution + brand-DNA fold-back) is M2, with its own risk budget (§2). Lago/Anima/Haima wiring is M2 (§3) — it is a weeks-long cross-subsystem integration, not a Week-1 task. TikTok public posting is gated on a manual audit milestone, not a code milestone. SECTION 08

Open questions

Six unresolved design decisions surfaced by the canon and research. None block the §00 gate decision (don't build yet); all must be answered before M1 Week 1.

  1. Generation subprocess vs native. Does kleos-generate shell out to the existing content-engine scripts as a subprocess (fast, reuses live Gemini wiring, process-isolated) or reimplement generation natively in Rust (tighter, but duplicates a working pipeline)? Subprocess is the dep-chain-cheap default; revisit only if latency or error handling forces it.
  2. TikTok audit gating. The content-posting audit is a manual, opaque, multi-week process. Does M1 ship the TikTok transport as SELF_ONLY drafts and submit for audit in parallel, or defer the TikTok rail entirely to M2? (Current plan: ship SELF_ONLY, audit in parallel — but the audit timeline is outside our control.)
  3. Where the LLM-controller runs. The "decide" step (what to generate next, which variant) needs an LLM call. Does it run inside the Railway daemon (cost, key custody in the cloud) or as an out-of-band call the daemon requests? Affects S-K1 spend attribution and the token-custody question below.
  4. Spend-attribution granularity. S-K1 caps daily spend — but at what granularity? Per-rail? Per-tenant? Per-asset? Per-variant (needed for M2's bandit reward)? The M1 store schema must record enough to support M2's attribution without a migration.
  5. Browser-automation fallback ToS tension. When a platform lacks a publish API (or gates it behind audit), browser automation is the fallback — but it violates most platform ToS and risks the very bans the compliance pillar sells against. Hard rule candidate: no browser-automation publishing rail ever, accept the API constraints as the design envelope. Needs a decision.
  6. Multi-tenant token custody. The Kleos-recursion endgame (sell the engine to operators who run it for their vertical) means custodying their platform OAuth tokens. Where do those live, how are they encrypted, and does Anima (in-flight) own identity here — or does M1 stay strictly single-tenant (Broomva's own accounts) until Anima is M1-ready?

Plus a seventh, from research: the cold-start bootstrap — the loop needs some measured retention signal before "decide" has anything to optimize. How does the first batch get seeded (manual operator picks? imported consumer-niche priors, clearly labeled per the KPI-honesty gate?) without claiming validated numbers that aren't?
